## Import Pipeline Design — The Rules

**Read `HobsonsNotes/import-pipeline-plan.md` first.** It has the full
architecture, import cycle, and all rules. What follows is the
reasoning and context behind those rules that might not be obvious from
the plan alone.

### Deduplication — Three Layers

This is the most important thing to get right. Dan pressure-tested this
extensively.

**Layer 1: Batch idempotency.**
`UNIQUE(ref_id, batch_id)` on every staging table. `INSERT ON CONFLICT
DO NOTHING`. Re-running a parser on the same file is a no-op. Mechanical
and boring. Don't overthink it.

**Layer 2: Ledger dedup.**
Before promoting ANY row from staging to the ledger, check
`journal_entry_reference` via `findNonVoidedByReference()`. If a JE with
that reference already exists, the transaction is already posted.

CRITICAL DISTINCTION:
- Native ref matches (Visa long numeric, SECU embedded ref, Synchrony
  P934200...): auto-mark as duplicate. These are reliable identifiers.
- Composite key matches (Fidelity history `account|date|amount`, Ally
  `account|date|amount`): ALWAYS flag for Dan's review. NEVER auto-skip.
  Two genuinely distinct transactions can produce the same composite key
  (e.g., two $1,000 Zelle payments on the same day). Dan said: "I'm not
  running a bank here" — he doesn't care about misattribution between
  identical transactions, but he wants to see them.

**Layer 3: Transfer pairing.**
Two-sided transactions don't promote until both sides are accounted for.
"Accounted for" means EITHER:
- Both sides present in staging, OR
- One side in staging, partner already in the ledger (check
  `journal_entry_reference` for the expected partner ref)

This is essential. Dan specifically corrected me on this — the original
plan only checked staging. But if one leg was posted manually or from a
previous batch, the other leg would sit in staging forever waiting for a
partner that's already gone. CHECK BOTH PLACES.

Posted as a single balanced JE. Reference: `{from_ref}|{to_ref}`.

### The Manual Test Strategy

Dan's idea, and it's good. Before running the full importer:

1. Post one real transaction per source directly to the ledger via CLI,
   with the proper `--ref` flag.
2. Run the parser on the full import file. It stages everything,
   including the one you already posted.
3. When promotion runs, that one row should be detected as already
   existing in the ledger (Layer 2 dedup).
4. This proves the dedup-against-ledger logic works with real data.

If the manually-posted transaction does NOT show up as a duplicate,
something is broken in the promotion logic. Fix it before proceeding.

### Import Cycle — Every Round

1. Parse source file into staging
2. Auto-categorise (merchant rules for expenses, action rules for
   Fidelity history)
3. Dedup check against ledger
4. Promote clean rows via CLI
5. Flag composite-key matches for review
6. Match transfer pairs (staging + ledger)
7. Dan and Hobson review EVERY remaining row. No exceptions.
8. Staging is empty when done. If not, something is wrong.

### Action-Based Rules

| Action | Taxable Brokerage (1210) | Property CMA (1110) | IRAs |
|---|---|---|---|
| DIVIDEND RECEIVED (SCHD etc.) | Dr cash, Cr 4200 | Dr cash, Cr 4200 | SKIP |

| REINVESTMENT (SPAXX) | SKIP | SKIP | SKIP |
| YOU BOUGHT / YOU SOLD | SKIP (portfolio) | N/A | SKIP |
| TRANSFERRED TO/FROM | TRANSFER pair | TRANSFER pair | SKIP |
| DIRECT DEPOSIT (payroll) | SKIP | N/A | N/A |
| DIRECT DEPOSIT (cash-back) | Dr cash, Cr 4200 | N/A | N/A |
| DIRECT DEBIT / BILL PAYMENT | Dr expense, Cr cash | Dr expense, Cr cash | N/A |
| Fees/Commission (non-zero) | Dr 7220, Cr cash | Dr 7220, Cr cash | SKIP |

**Key context:**
- SPAXX IS the cash inside the brokerage. All dividends land in SPAXX
  first. The REINVESTMENT row that follows is just SPAXX buying more
  SPAXX — it's a non-event.
- IRA transactions ALL skip. Dan never withdraws. Portfolio module
  tracks positions.
- Property CMA dividends are Other Revenue (4200), NOT rental income
  (4110). Per GAAP: interest on an operating account is other revenue.
- Payroll direct deposits skip. Dan only tracks net deposits arriving,
  not paycheck line items.
- "ELAN CARDSVCRedemption" is Fidelity Visa cash-back rewards — that's
  revenue (4200).

### No native reference IDs

The Fidelity history CSV has NO transaction reference numbers. Composite
key: `{account_number}|{date}|{amount}`. These always go to review
lane during dedup (Layer 2 rule).

### CSV quirks
- Two blank lines at the top of the file
- Fidelity legal disclaimer footer (starts with "The data and
  information in this spreadsheet...") — strip everything after the
  last data row
- Settlement Date is often empty (cash transactions) — don't use it as
  part of the key
- Quantity field: `0.000` for cash transactions, real values for trades
- Multiple files may cover overlapping date ranges — the batch_id
  (filename) distinguishes them in staging

## Special Transaction Rules — GAAP Decisions

These were decided with Dan this session. They're in the pipeline plan
but here's the reasoning:

### Auto loan payment (Ford → 2230)
Whole payment posts as transfer: Dr 2230, Cr cash. No principal/interest
split — Ford doesn't break it out on the bank statement. Could do a
year-end adjustment with amortisation schedule if Dan wants.

### Personal mortgage (SECU → 5311)
Lump payment (principal + interest + escrowed tax + insurance) posts
whole to 5311. GAAP would want these separated but we don't have the
breakdown from the statement. Year-end adjusting entry to reclassify
principal as liability reduction on 2210 is optional.

### Merchandise refunds
Reduce the original expense account per GAAP. A grocery refund credits
5350, it does NOT post as revenue to 4200. Match refund to expense
where possible, flag for review when ambiguous.

### Credit card payments
Balance sheet transfer. Dr 2220 (Visa liability), Cr 1220 (SECU cash).
The expense was already booked when the purchase hit the card. The
payment just moves the obligation.

### Fidelity Visa paid from SECU
Not from Fidelity CMA. SECU checking is the payment hub. The payment
shows in both the Visa CSV (as a credit) and the SECU statement (as a
debit). Import from whichever source processes first; dedup catches it
on the second.
