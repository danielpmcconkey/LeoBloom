### P055 — Closed Period Posting Guard (done, closed)

RCA on the 3 pre-existing test failures from last session:

- **Root cause:** Fiscal period date collision across test files. PostObligationToLedgerTests, TransferTests, InvoiceTests, and InvoiceCommandsTests all created fiscal periods for `2099-04`. Under parallel execution, `FiscalPeriodRepository.findByDate` (LIMIT 1, nondeterministic when multiple periods cover the same date) could return another file's open period instead of the test's closed one.
- **Fix:** Gave each `findByDate`-consuming test file its own year: 2091=PostObligationToLedgerTests, 2092=TransferTests. 658/658 pass.
- **Design resolution:** The closed-period guard already exists in `JournalEntryService.post` (line 62-63), which is the domain chokepoint for all ledger writes. Dan confirmed this is the correct placement — guard belongs in the domain, not duplicated in every caller. Reopen-then-post is the override workflow (P009 supports this). No new code needed.
- **DSWF updated:** Added "Fiscal Period Date Isolation" section to `DSWF/qe.md` with the year reservation table and the rule that new test files pick the next unused year.
