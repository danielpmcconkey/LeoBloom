### The real problem: test architecture
Dan pointed me to `/workspace/PersonalFinance/Lib.Tests` — ~1168 tests, ~679 of which
hit the database, entire suite runs in 13 seconds. LeoBloom has ~112 DB tests and takes
10+ minutes with cascading timeout failures.

**PersonalFinance**: EF Core `DbContext` manages connection pooling automatically. Tests
read/write through the framework. No manual connection lifecycle.

**LeoBloom**: Raw Npgsql. Every test manually opens a connection, begins a transaction,
runs N SQL commands, rolls back in a `finally` block, then closes/disposes. If any test's
cleanup fails or if tests run in parallel, connections leak and subsequent tests wait 30
seconds each to timeout. One bad test poisons the entire suite.

**Dan's verdict: this is priority #1.** Fix the test architecture before finishing 007.

## What needs to happen

### 1. Design the new test connection strategy
Options to research:
- **Connection pooling with Npgsql**: `NpgsqlDataSource` (singleton) with pooling config.
  Tests should get connections from a shared pool, not open raw connections.
- **xUnit fixtures**: `IClassFixture<T>` or `ICollectionFixture<T>` for shared DB setup.
  One connection/transaction per test class instead of per test.
- **Respawn or test database reset**: Instead of transaction-per-test rollback, use
  Respawn to reset state between tests.
- **EF Core**: Probably overkill for LeoBloom's raw SQL approach, but worth discussing.

Key constraint: LeoBloom's tests are TickSpec/Gherkin BDD tests with step definitions.
The context record pattern (`BalanceContext`, `PostContext`, etc.) carries state through
steps. Whatever connection strategy we pick needs to work with that pattern.

### 2. Validate with Dan before implementing
This is an architecture change. Don't just start coding. Talk it through. Dan will have
opinions about which approach fits LeoBloom's style.
