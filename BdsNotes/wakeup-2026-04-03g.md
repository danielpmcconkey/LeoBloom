### TickSpec Ambiguity Tooling — NEW
Dan asked me to institutionalize the TickSpec step collision fix since I hit it on two consecutive projects.

**What was created:**
1. `Scripts/check-tickspec-ambiguity.sh` — extracts all step regexes from `*StepDefinitions.fs` files, flags exact duplicates and prefix overlaps (greedy `.+` traps). Includes "how to fix" guidance in output.
2. Claude Code hook in `.claude/settings.local.json` — `PostToolUse` on `Edit|Write`, runs `.claude/hooks/check-tickspec-ambiguity.sh`. The hook script bails immediately unless the file ends in `StepDefinitions.fs`, then runs the checker. Exit 2 blocks the action.
3. `.git/hooks/pre-commit` — also runs the checker when `*StepDefinitions.fs` files are staged.
4. Memory: `feedback_tickspec_step_ambiguity.md`
5. Research doc: `BdsNotes/tickspec-step-patterns.md`

## Lessons learned this session
- TickSpec step regex ambiguity is a recurring trap. Now institutionalized with checker + hook.
- Don't launch multiple `dotnet test` runs in background — connection pool exhaustion cascades into 30-second timeouts per test, poisoning the whole suite.
- PO caught the `account_code` input gap on first review — backlog said "account_id (or account_code)" and I only implemented account_id. Read the backlog inputs line carefully.
