### Design discussion → P052

Discovered the real problem: `ledger.account` has no sub-type classification. GAAP needs it. Every real accounting package (GnuCash, QuickBooks, ERPNext, hledger) has either a sub-type enum or promotes cash to a first-class type.

Research findings:
- **GnuCash:** flat enum with 15 types (BANK, CASH, ASSET are peers)
- **QuickBooks:** 3-level system (classification → type → detail type). Cash flow uses type = "Bank"
- **ERPNext:** 2-level (root_type + account_type with ~32 sub-types)
- **hledger:** flat enum where Cash is an explicit subtype of Asset

Dan + Hobson chose: **QuickBooks/ERPNext-style two-level system.** New nullable `account_subtype varchar(25)` column on `ledger.account`. F# DU polices valid values. No lookup table, no FK.

### P052 — Account Sub-Type Classification (planned, not started)

Plan written at `Projects/Project052-AccountSubTypes/Project052-plan.md`. Key decisions:

**Subtypes by account type:**
- Asset: Cash, FixedAsset, Investment
- Liability: CurrentLiability, LongTermLiability
- Equity: none (null only — defer until reporting needs it)
- Revenue: OperatingRevenue, OtherRevenue
- Expense: OperatingExpense, OtherExpense

**Design rules:**
- Nullable — headers get null, leaves get subtypes where meaningful
- Flat DU + validation function (`isValidSubType`) on every write path
- No equity subtypes yet — redundant until there's a reporting need
- Brokerage (1210) = Investment, not Cash
- Subtype is a label, no semantic weight like normal_balance

**Blast radius:** Migration, Domain types, every Ledger repo/service that reads/writes accounts, Ops (if it reads accounts), all tests that create accounts, seed data.
