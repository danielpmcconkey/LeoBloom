### What Dan and BD decided about BDD flow

**This is the big one. Read the memory file:**
`~/.claude/projects/-workspace/memory/project_leobloom_bdd_flow.md`

Summary: LeoBloom treats specs as the product and code as disposable. The rebuild
manifest is: BRD + BDD doc + feature files. IDs tie them together:

- **BDD IDs** — project-scoped (`BDD-001`), ephemeral, live in project dir
- **Feature IDs** — global (`@FT-LSC-001`), permanent, tagged on Gherkin scenarios
- **Test-results** — one-time map from BDD IDs → Feature IDs per project

This flow starts with Project 3.

## Things to argue about next session

1. **Should the `Documentation/` dir follow proofmark's pattern exactly?** Proofmark
   has `README.md` (routing table), `overview.md`, `cli.md`, `configuration.md`,
   `testing.md`, plus module-level docs. LeoBloom doesn't have enough code yet to
   justify module docs. Maybe just `overview.md` and `testing.md` for now.

2. **Do we retroactively write a BDD doc for Project 1 and 2?** The BDD flow didn't
   exist when those projects shipped. The feature files exist but there's no formal
   BDD doc with `BDD-*` IDs. Options: write them retroactively for completeness,
   or just start clean from Project 3 and accept that 1-2 are pre-flow.

3. **Project 2 test-results need rewriting** once FT tags exist. The current
   test-results files don't have FT IDs because we hadn't invented them yet.

4. **Should `Specs/` be at repo root or under `Src/`?** The feature files are test
   infrastructure, which argues for `Src/`. But they're also the rebuild manifest,
   which argues for repo root visibility. Proofmark's `Documentation/` is at root.
   I lean root.
